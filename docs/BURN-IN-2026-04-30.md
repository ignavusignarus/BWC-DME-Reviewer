## P0 — production blockers

### 1. BWC video sources play as audio (and 415 on every request) — `editor/components/reviewer/MediaPane.jsx:8`

```js
const isVideo = source.mode === 'video';
```

The manifest emits `mode: 'bwc' | 'dme'` (see `engine/project.py:60-76`). `'video'` is never a value. So **for every BWC source**:

- `<video>` is never rendered.
- The `<audio>` fallback gets `src="/api/source/audio?…mp4"`.
- The engine returns 415 (`source is video; use /api/source/video`, `engine/server.py:436`).
- `<audio>.error.code = 4 (MEDIA_ERR_SRC_NOT_SUPPORTED)`, transport stays at `00:00 / 00:00`.

`Waveform.jsx` fetches the same audio URL and fails decode silently.

**Fix:** `const isVideo = source.mode === 'bwc';` (or extension-based: `VIDEO_EXTENSIONS.has(ext)`).

**Repro:** any BWC `.mp4` source. Verified on `sja00161_…mp4` (12:31 long): pipeline completed cleanly, transcript loaded, timeline rendered — but no media element to play.

### 2. Media URLs are relative — broken under `file://` — `MediaPane.jsx:10-11`, `Waveform.jsx`

```js
const audioUrl = `/api/source/audio?${params}`;
const videoUrl = `/api/source/video?${params}`;
```

Production loads `index.html` from `file://`. A relative URL `/api/source/audio?…` resolves to `file:///api/source/audio?…` and 404s. **Media playback can't be working in the desktop app today.**

The harness only worked because `dev_server.py` serves the page from `http://127.0.0.1:8765` and proxies `/api/*` to the engine. In Electron there is no such proxy.

**Fix:** prefix with `getBaseUrl()` like `apiGet`/`apiPost` do (`editor/api.js:11-20`). E.g. `${base}/api/source/audio?${params}`.

**Note:** the Electron preload bridge already exposes `getEngineUrl()` so the renderer has the engine origin. The fix is one line per URL.

### 3. Layout grid collapses transcript off-screen on long audio — `ReviewerView.jsx:186`

```jsx
<div style={{ flex: 1, display: 'grid', gridTemplateColumns: '1fr 360px', minHeight: 0 }}>
```

`1fr` doesn't include `min-width: 0`, so when the left column's intrinsic min-content (the collapsed timeline) is wider than the viewport, the column expands to fit and the right column (transcript + context-names) is rendered at e.g. `x = 16985px`. Visible result: empty middle of viewport, no transcript or context inputs.

For the 59-min ENT audio the rendered grid was `16984px 360px`; transcript column at x=16984.

**Fix:** `gridTemplateColumns: 'minmax(0, 1fr) 360px'`.

### 4. TopBar source-switcher dropdown is empty — `TopBar.jsx:15`

```js
const completedFiles = manifest.files.filter(f => f.completed);
```

The manifest never sets `completed` on any file (see `engine/project.py:96-139`). So `completedFiles` is always `[]` and the `<select>` renders zero options. Visually the dropdown is just a tiny chevron with no items inside — looks broken / placeholder.

Same broken assumption: `EditorApp.jsx:50` (`if (… || file.completed) …`) — the manifest-driven fast path to reviewer never fires.

**Fix (option A, smaller):** TopBar fetches each file's status on mount and filters by completed. **Option B (cleaner):** engine populates `files[i].completed` from `pipeline-state.json` in `open_project`. Option B also fixes EditorApp's fast-path.

### 5. All segment IDs are 0 — auto-scroll completely broken — `engine/pipeline/transcribe.py:104`

```python
"id": int(s.id),
```

faster-whisper's `Segment.id` is **always 0 for VAD-filtered output** (we use `vad_filter=True`). I verified on the ENT transcript: 1225 segments, **`unique(id) = {0}`**.

Downstream impact:

- React renders 1225 elements with `key={s.id}` = duplicate-key warnings, plus reconciliation thrash.
- `TranscriptPanel.jsx:46` does `querySelector('[data-segment-id="${segments[active].id}"]')` to scroll the active segment into view. Since every segment shares id=0, the selector always returns the first segment. **Confirmed empirically:** seek to 25:00 → active segment correctly highlighted at y=19177, but `panel.scrollTop = 0` (auto-scroll did not move).

**Fix:** assign sequential IDs in `_build_transcript`:

```python
for i, s in enumerate(segments_with_words):
    enriched = {"id": i, ...}
```

Same fix needed in `transcribe.py:101-111` if you want IDs from the start.

### 6. Re-transcribe with context names degrades transcript — `transcribe.py:196-201`

After typing `Belafsky / Wojtaszewski / Ravikudi` into the names panel and clicking "Apply & re-transcribe", the new transcript inserted **"Hello, everyone."** eleven times in succession at 00:15-00:25, replacing what was previously a coherent passage.

The initial_prompt assembly is:

```
This is audio from a recorded interaction. Names mentioned may include: Belafsky, Wojtaszewski, Ravikudi.
```

That phrasing isn't obviously hallucination-bait, but combined with `vad_filter=True` and the current decoder params it triggered a hard repetition loop. Whether the right fix is:

- a different prompt format (e.g., drop the `"This is audio from..."` framing — Whisper prompts are stylistic, and this preamble may be biasing toward greeting language), or
- raising/tightening `compression_ratio_threshold` (currently 2.4) to catch the repetition, or
- a separate `repetition_penalty` / `no_repeat_ngram_size` style guard,

is worth discussion. **What's clear:** the feature can produce a worse transcript than the no-prompt baseline. That's a regression visible to users.

## P1 — bugs, doesn't block

### 7. Pipeline completion via polling doesn't auto-route to reviewer — `EditorApp.jsx:107-128`

`startPolling` stops when `!isActiveStatus(resp.status)`, but never calls `setView('reviewer')`. So after the user clicks an unprocessed file, watches the pipeline run, sees the green ✓, they have to **click the file a second time** to enter the reviewer.

The "fast path" at `EditorApp.jsx:73-86` does route to reviewer when the initial `/api/source/process` POST returns `completed`, but that path only fires when the source was already cached at click time.

**Fix:** in the polling completion branch, set `reviewSource` and `view='reviewer'`, mirroring the fast path.

### 8. Engine returns 501 for `OPTIONS` — CORS preflight broken — `engine/server.py`

`BWCRequestHandler` has `do_GET` and `do_POST`; no `do_OPTIONS`. Any browser client on a different origin sending a non-simple request (e.g., `Content-Type: application/json` POST) gets a 501 preflight and the actual request is never sent.

Production Electron loads `file://` so this rarely matters, but this is what blocked my Chrome harness initially. Also future-proofing: any external automation client that talks to the engine over the network will hit this.

**Fix:** add a `do_OPTIONS` that returns 204 + the same CORS headers `_send_json` already emits.

### 9. Collapsed timeline can be 17,000+ pixels wide — `CollapsedTimeline.jsx`

24 px per silence cell × hundreds of silence cells in a 60-min file = absurd intrinsic width. With the grid fix from P0 #3 this stops blowing out the layout, but the timeline still overflows its container and either needs `overflow-x: auto` on the wrapper or smaller `SILENCE_PX_DEFAULT` for long files.

(The uncompressed timeline mode uses `widthPct` and fits cleanly — that mode is fine.)

## Harness notes (for the next session)

`dev_server.py`, `dev-chrome.html`, `dev-shim.js` are in the repo root. To re-launch:

```bash
.venv/Scripts/python.exe serve.py &
# capture port from "BWC_CLIPPER_PORT=NNNNN"
.venv/Scripts/python.exe dev_server.py --engine-port NNNNN --listen-port 8765 &
# open http://127.0.0.1:8765/dev-chrome.html?engine=NNNNN&folder=C:/path/to/folder
```

The `?folder=` query param feeds straight into `pickFolder()`. The dev banner at the top of the page shows the engine port and current folder.

These three files are dev-only and not loaded by Electron (which still uses `index.html`).

The `BWC_CLIPPER_FFMPEG_DIR` env var must be set when launching `serve.py` outside of Electron — Electron normally sets it from its ffmpeg-downloader. Use:

```
export BWC_CLIPPER_FFMPEG_DIR="C:/Users/<you>/AppData/Roaming/bwc-clipper/ffmpeg"
```
